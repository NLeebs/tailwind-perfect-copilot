## Milestone Summary

### Key Accomplishments

1. **Shared infrastructure foundation** — CodeCallout RSC, cn() utility, dark mode fixed to @custom-variant dark exclusively, all runtime deps (clsx, tailwind-merge@^3, motion, shiki, shiki-magic-move) installed
2. **Slide 2 — What Is Tailwind** — ButtonComparison island establishing semantic-vs-utility contrast; "class list is documentation" pattern
3. **Slide 3 — Core Utility Classes** — CardBuilder 6-step progressive demo (STEPS lookup table) + FlexGridComparison RSC; TDD with Vitest
4. **Slide 4 — Responsiveness & Dark Mode** — ResponsiveDemo tab island with LAYOUT_CLASSES map; dark: prefix panel; stacked dark:md:hover: variant demo
5. **Slide 5 — Customizing Tailwind** — ShikiBlock async RSC with Vitest mock; @theme token → utility pipeline; @utility scrollbar-hidden; @layer base h1 live demo
6. **Slide 6 — Conditional Styling** — ConditionalPanels + DataActiveDemo client islands; peer-invalid form; complete Vitest test suite
7. **TV readability and quality pass** — Zero text elements needed 3xl: escalation (all pre-compliant); reduced-motion gate confirmed; build/lint green

### Patterns Established

- **Single-source const pattern** — `className` string extracted as a named const so both the element and its `<CodeCallout>` use the same value (prevents drift)
- **Lookup-table map for toggled state** — complete static class strings keyed by state value; no string interpolation
- **Leaf-node client islands** — interactive demos isolated in `"use client"` components; `page.tsx` stays RSC
- **Vitest mock alias pattern** — `resolve.alias` in `vitest.config.mts` for WASM-heavy or async RSC libraries (shiki, ShikiBlock)

### Key Decisions

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Tailwind v4 config in globals.css only | Demonstrates v4's CSS-first config approach — teaches by example | ✓ Good |
| Class-based dark mode via @custom-variant | Shows how Tailwind v4 handles custom variants; works alongside ThemeToggle | ✓ Good |
| Home NavCard grid as navigation hub | Simpler to navigate for presenter and audience; avoids slide-runner complexity | ✓ Good |
| Minimal inline callouts over full code blocks | Audience at TV distance; dense code blocks hurt readability | ✓ Good |
| Per-slide RSC + "use client" isolation | Teaches the RSC boundary model implicitly while keeping demos interactive | ✓ Good |
| tailwind-merge@^3 (not v2) | v2 is Tailwind v3-only; project uses Tailwind v4 | ✓ Good |
| motion package (not framer-motion) | Maintained package; imported as "motion/react" | ✓ Good |
| CodeCallout class string as single static literal | Tailwind purger cannot detect dynamically constructed class strings | ✓ Good |
| !important in .dark .shiki CSS override | Shiki dual-theme embeds inline style= that beats class selectors | ✓ Good |
| ShikiBlock Vitest mock via resolve.alias | Async RSC cannot render in jsdom without Next.js framework | ✓ Good |
| ESLint globalIgnores for worktree .next dirs | Parallel agent .next/ generated files caused spurious lint errors | ✓ Good |
